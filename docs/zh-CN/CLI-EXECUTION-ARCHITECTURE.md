# Obsidian CLI 执行通道与 ObsidianHub 架构关系

## 1. 目的

本文档用于明确：

- ObsidianHub 与 Obsidian 官方 CLI 的关系
- 客户端 Obsidian CLI 在架构中的角色
- 为什么 CLI 执行通道是补充路径，而不是服务端主架构替代
- 它如何与 Sync、Git、快照、agent 控制平面对接

## 2. 核心结论

ObsidianHub 的主架构仍然以**服务端持有完整 vault 与控制平面**为基础。

Obsidian CLI 不应成为服务端运行前提，但可以成为一个非常有价值的：

- **客户端原生命令执行通道**
- **更贴近官方语义的执行器**
- **agent 受控操作的补充路径**

## 3. 两类执行通道

### A. 服务端原生执行通道
由 ObsidianHub 服务端直接操作 vault。

适合：
- 后台任务
- 服务器常驻 agent
- 无客户端在线时
- 批量整理
- 索引、备份、恢复、Git checkpoint 等基础设施级任务

### B. 客户端 CLI 执行通道
由本地安装了 Obsidian 的客户端，通过 Obsidian CLI 执行命令。

适合：
- 用户本地交互式操作
- 更贴近 Obsidian 官方语义的 note 操作
- 可命令化、可审计、可约束的 agent 行为
- 某些高层操作下的 token 节省

## 4. 为什么 CLI 通道有价值

### 4.1 更贴近官方语义
CLI 由 Obsidian 官方提供，因此某些操作的行为天然更接近桌面 Obsidian 的实际语义。

### 4.2 更可控的 agent 操作面
相比直接给 agent 文件系统权限，CLI 通道可以提供更窄、更明确的能力边界。

### 4.3 某些场景可能更省 token
当高层操作可直接映射为命令时，agent 不需要自己读大量文件再推理如何编辑。

### 4.4 有助于降低服务端语义落后风险
CLI 可以作为官方能力基线，减少服务端在某些本地语义上的偏差。

## 5. 为什么不把 CLI 当主架构

尽管 CLI 很有价值，但它不适合作为 ObsidianHub 的唯一主执行路线。

原因：
- 依赖某个本地客户端在线
- 依赖本地安装与运行 Obsidian
- 不适合作为服务器常驻自动化的唯一基础
- 多客户端场景会出现“由谁执行”的协调问题
- 无法替代服务端的 Git、快照、统一控制平面职责

## 6. 推荐架构：双执行通道

ObsidianHub 推荐采用：

### 控制平面在服务端
服务端负责：
- 权限
- 调度
- 日志
- 审计
- Git checkpoint
- 快照/恢复
- agent 管理

### 执行器有两种
1. **Server Executor**：直接改服务端 vault
2. **Client CLI Executor**：通过本地 Obsidian CLI 执行

## 7. 路由原则

未来执行某个任务时，可根据以下因素决定路由：

- 当前是否有可用客户端在线
- 任务是否更适合官方语义执行
- 是否属于基础设施任务（更适合服务端）
- 是否需要更窄的命令能力边界
- 是否需要在无客户端情况下也能执行

## 8. 与其他模块的关系

### 与 Sync 的关系
CLI 通道不替代 Sync API。Sync 负责工作态传播；CLI 通道负责某些本地语义更强的执行任务。

### 与 Git 的关系
CLI 通道不替代 Git checkpoint。CLI 执行结果仍应回流服务端，由服务端决定是否形成 checkpoint。

### 与快照/恢复的关系
即使操作通过客户端 CLI 执行，服务端仍应拥有最终的快照与恢复控制能力。

### 与 agent 控制平面的关系
agent 不应直接无约束调用本地 CLI。更合理的方式是由 ObsidianHub 控制平面决定何时将任务路由到 CLI 执行器。

## 9. 当前建议

当前阶段不建议立即实现完整 CLI 执行器，但建议：

1. 在架构层明确保留该通道
2. 在文档中定义它的角色与边界
3. 后续在轻插件或客户端连接层中预留接入点

## 10. 一句话定位

**Obsidian CLI 在 ObsidianHub 中最适合被定义为：客户端原生命令执行通道，而不是服务端主架构依赖。**
