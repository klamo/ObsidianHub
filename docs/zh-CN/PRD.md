# ObsidianHub 产品需求文档（PRD）

## 1. 产品概述

ObsidianHub 是一个面向个人用户的自托管 Obsidian Server，核心目标是为个人 vault 提供：

- 远程访问
- 多设备工作态同步
- Git 优先的版本管理
- 多 agent 受控编辑
- 可审计、可回滚、可恢复的服务层

它不是 Obsidian 替代品，也不是团队知识库平台。

## 2. 目标用户

### 核心用户
- 使用 Obsidian 作为个人知识库的用户
- 有 VPS / NAS / 家庭服务器的自托管用户
- 希望让 AI / automation 参与笔记整理、索引、发布的高级用户
- 希望继续在本地 Obsidian 中工作，但想通过自己的服务器同步与管理数据的用户

### 非目标用户
- 需要企业协作权限体系的团队
- 需要实时多人协作编辑的组织
- 想用它替代完整在线文档平台的用户

## 3. 用户问题

1. 不在本机时，访问 vault 不方便
2. 多设备之间缺少统一、可控的同步路径
3. 多个客户端 / 进程 / agent 同时操作 vault 时，容易产生冲突或覆盖
4. 现有同步方案通常不提供面向 agent 的受控 API
5. 让 agent 直接碰文件系统风险太高
6. 自动化改坏笔记后，缺少回滚与审计能力
7. 用户希望像实时文档一样顺滑工作，但又希望保留 Git 的版本管理能力

## 4. 产品目标

### 必须达成
- 用户可以远程访问并查看自己的 vault
- 用户可以继续在本地 Obsidian 中工作，并通过客户端接入 ObsidianHub
- 当前工作状态可以在多设备间同步
- Git 用于保留更稳定、更有意义的版本节点
- agent 可以通过受控 API 操作笔记
- 关键写入具备冲突保护与恢复能力
- 部署与初始化足够简单，适合 Docker 场景

### 本阶段不追求
- 团队协作
- 复杂权限模型
- 重型实时协作编辑
- CRDT / OT
- 大而全平台化

## 5. 核心场景

### 场景 A：远程查看与搜索
用户出门在外，通过浏览器查看笔记、搜索内容、下载附件。

### 场景 B：本地 Obsidian 持续工作
用户继续在自己的 Obsidian App 中编辑，连接自己的 ObsidianHub 服务，与其他设备同步当前工作状态。

### 场景 C：轻插件连接服务端
用户通过轻量 Obsidian 插件配置服务器地址、凭据、同步状态，并触发基础 sync 操作。

### 场景 D：agent 整理知识库
整理 agent 通过受控 note actions 在授权目录中创建笔记、更新正文、补 frontmatter、重命名与归档笔记。

### 场景 E：agent 建立索引
索引 agent 读取全库内容，生成 tags、links、索引文件，但不能随意改正式笔记。

### 场景 F：错误恢复
某个 agent 或用户误改了大量内容，系统可基于快照、Git checkpoint 或冲突副本恢复。
