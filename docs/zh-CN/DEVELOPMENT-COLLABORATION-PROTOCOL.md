# ObsidianHub 开发协作执行方案（v1）

## 1. 目的

本文档定义 ObsidianHub 在当前阶段的默认开发协作方式，用来解决三个问题：

1. 避免“写一点、停一下、再问下一步”导致推进粒度过碎
2. 明确哪些决策可以由 Moss 直接推进，哪些必须先与 Klamo 确认
3. 让编码执行层（Moss / Codex / 后续专职 coding agent）接到的是可施工任务包，而不是模糊方向

这份文档不是产品 PRD，也不是纯技术架构文档。

它是：
- 项目级执行协议
- 任务流转规则
- 默认决策边界说明
- 质量与交付标准

---

## 2. 当前推荐协作模型

当前阶段采用：

**Moss 主控 + 按任务粒度选择编码执行层**

具体分工如下：

### Moss 的默认职责
- 产品目标收敛
- 技术方案收敛
- 阶段目标定义
- 任务拆解与排序
- API / schema / module boundary 设计
- 给 Codex 或 coding agent 产出结构化任务包
- 审阅实现结果
- 运行 typecheck / test / 基础验收
- 更新文档、路线图与阶段状态
- 在必要时亲自完成小范围代码改动

### 编码执行层的默认职责
编码执行层可以是：
- Moss 自己
- Codex
- 后续独立的专职 coding agent

其核心职责是：
- 按任务包施工
- 完成明确范围内的代码修改
- 补齐必要测试
- 汇报实现结果、风险与未完成项

### 当前默认选择规则
- **小改动 / 局部修复 / 低上下文负担任务**：Moss 可直接完成
- **中等复杂度 / 重复性较强 / 可清晰验收的实现任务**：优先交给 Codex
- **并行编码任务明显增多时**：再引入独立 coding agent 承担长期施工

结论：

> 当前 ObsidianHub 不应默认“始终由 Moss 自己写”，也不应急着把协作方法先做成 skill；更优先的是先稳定项目级执行协议。

---

## 3. 为什么需要这份协议

当前容易出现的一种低效模式是：

1. 先做一小段
2. 做完后暂停
3. 再询问方向
4. 再做下一小段

这种模式的问题不是“代码写不动”，而是：
- 决策被切得太碎
- 任务包太小
- 编码层无法连续推进
- 主控层被拖入过多实现细节
- 用户需要频繁充当即时路由器

因此，ObsidianHub 当前阶段应该改用：

**先前置收敛，再连续推进。**

也就是：
- 先写清当前阶段目标与技术约束
- 再定义一轮可施工任务包
- 编码层连续完成
- Moss 做统一验收与收口
- 只在真正跨边界时再停下来确认

---

## 4. 文档分层

为避免 PRD、架构、执行规则混杂，当前文档应分三层：

### 第一层：产品层文档
用于回答“做什么、为什么做”。

包括：
- `PRD.md`
- `MVP.md`
- `README.md`

### 第二层：技术收敛文档
用于回答“系统怎么分层、怎么约束”。

包括：
- `ARCHITECTURE.md`
- `TECH-STACK.md`
- `SYNC-AND-VERSIONING.md`
- `NOTE-ACTIONS-AND-TOOL-CONTRACT.md`
- 其他专题设计文档

### 第三层：执行层文档
用于回答“现在这一阶段怎么推进”。

包括：
- `DEVELOPMENT-ROADMAP.md`
- 本文档 `DEVELOPMENT-COLLABORATION-PROTOCOL.md`
- 未来如有需要，可补 `TASK-PACKS/` 目录

---

## 5. 默认推进节奏

默认按以下节奏工作：

### Step 1：阶段收敛
Moss 先明确：
- 当前正在推进哪个 Phase
- 这一阶段的直接目标
- 当前不做什么
- 验收标准是什么

### Step 2：形成一轮任务包
Moss 把本轮工作写成结构化任务包，至少包含：
- 目标
- 修改范围
- 不可改动范围
- 接口约束
- 测试要求
- 完成定义

### Step 3：编码执行
由 Moss / Codex / coding agent 在不越界的前提下连续施工。

### Step 4：验收与收口
Moss 负责：
- 阅读 diff
- 检查边界是否漂移
- 运行 typecheck / test
- 必要时补小修
- 更新文档与阶段状态

### Step 5：再决定下一轮
只有当以下情况出现时，才需要再向 Klamo 询问方向：
- 遇到架构分叉
- 需要新增显著范围
- 需要改变已定产品边界
- 验收过程中发现当前方案不合理

---

## 6. 默认决策权限

## Moss 可直接决定的事项
以下事项默认可由 Moss 自行判断并推进，事后写入技术文档或变更说明即可：

- 目录内局部实现细节
- 文件拆分方式
- 类型命名微调
- helper / utility 的组织形式
- route / handler 内部结构
- 测试文件布局
- 小范围重构
- 不改变对外契约的内部抽象调整
- 文档补充与结构优化
- 当前 phase 内的子任务排序

## 必须先确认的事项
以下情况应先与 Klamo 确认：

- 产品目标或非目标发生变化
- MVP 范围扩大
- 对外 API 契约明显变化
- 引入新的核心依赖或基础设施
- 从轻量路线切向更重方案（如引入复杂实时协作、复杂权限模型）
- 影响部署方式、成本或长期维护方式的决策
- 需要删除或推翻已完成的大块实现
- 需要对外发布、公开同步或影响第三方仓库/服务的动作（若未明确授权）

## 默认可“先做后报”的原则
若某项变更满足以下条件，可默认先做后报：
- 不改变已定产品方向
- 不扩大当前 phase 范围
- 不改变对外 contract
- 可被 typecheck / test / 文档验证
- 回滚成本低

---

## 7. 技术执行 PRD 的最小模板

当一个阶段即将进入连续开发时，Moss 应先补齐对应的“技术执行 PRD”要点。未必单独成文件，但至少要覆盖以下内容：

1. **目标**
   - 这轮想打通什么
2. **边界**
   - 本轮只动哪些模块
   - 哪些模块暂时不碰
3. **接口**
   - 输入 / 输出 / 错误语义
4. **实现约束**
   - 是否必须复用已有包
   - 是否禁止越过 actions 直连底层
5. **测试要求**
   - 至少需要哪些测试
6. **完成定义**
   - 怎样才算这轮真正完成

说明：
- 技术执行 PRD 不追求面面俱到
- 它的目的不是增加文档负担
- 它的目的是让编码层拿到后可以连续施工，而不是边猜边写

---

## 8. 任务包模板

建议未来每一轮编码任务尽量遵循如下模板：

### 任务名
一句话描述本轮目标。

### 背景
为什么做这轮。

### 目标
本轮完成后应该打通什么。

### 修改范围
- 允许修改：
- 尽量不改：
- 明确不改：

### 实现要求
- contract 约束
- 错误处理要求
- 命名与组织要求
- 是否需要复用现有模块

### 测试要求
- typecheck
- 单测 / 集成测试
- 最小端到端验证

### 完成定义
满足哪些条件才可视为 done。

### 交付回报
执行层需要说明：
- 实际改了什么
- 哪些地方和原计划不同
- 当前还剩什么风险

## 9. 编码执行层使用规则

## 何时由 Moss 直接写
适用于：
- 10–60 分钟内可完成的小任务
- 需要强语义判断的局部边界调整
- 文档与代码紧耦合的小改动
- 验收过程中发现的收口补丁

## 何时优先调用 Codex
适用于：
- 明确目录范围内的 scaffold / wiring
- 重复性的 handler / schema / test 补齐
- 基于现有 contract 的批量实现
- 结构化任务包足够清晰、可直接验收的任务

## 何时考虑独立 coding agent
当出现以下信号时，可以考虑升级到独立 coding agent：
- 持续有并行编码需求
- 前后端任务同时推进
- 单轮实现量明显增大
- 需要更稳定的长期编码线程
- Moss 在审阅、测试、文档、任务管理上的负担开始高于合理范围

当前判断：

> ObsidianHub 还没到必须先做独立 coding agent 的程度，但已经适合采用“主控 + 可外包执行层”的模式。

---

## 10. 质量门槛

默认情况下，任何一轮代码提交前都应尽量满足：

- workspace `typecheck` 通过
- 相关 package 测试通过
- 新增能力至少有最小验证
- 不绕开已定义的 contract / action layer
- 文档在必要处同步更新
- commit 粒度清晰，可被回顾

如果某轮只是文档 / 方案提交，则至少满足：
- 能清楚约束下一轮施工
- 不与现有架构文档冲突
- 可以被后续任务包直接引用

---

## 11. 当前阶段的直接应用（Phase 3）

基于当前项目状态，这份协议在最近一轮应落到：

### 当前主目标
在 `apps/web` 中打通 note actions 的最小 HTTP / API handler。

### 当前默认边界
- 优先复用 `packages/actions`
- 不绕开既有 note action contract
- 先做最小统一入口，不急着铺开复杂认证系统
- 先打通 validate → execute → format response
- 先覆盖 6 个核心 note actions

### 当前建议分工
- Moss：确定 handler 形状、response envelope、错误边界、任务包与验收标准
- Codex 或执行层：完成 route wiring、基础 formatter、测试补齐
- Moss：验收、跑 typecheck/test、更新文档并提交

---

## 12. 与 skill 的关系

这份方案当前**先不做成 skill**。

原因：
- 它首先是 ObsidianHub 的项目执行协议
- 当前最重要的是让项目推进速度提高
- 不是先把方法抽象得很漂亮

何时适合升级为 skill：
- 这套方法在 ObsidianHub 中稳定跑过多轮
- 未来还会在其他代码项目复用
- 已经沉淀出稳定模板、触发条件和边界规则

到那时，可以把它抽象成类似：
- `project-execution-protocol`
- 或 `pm-coding-orchestration`

但现在不必提前抽象。

---

## 13. 当前默认结论

对于 ObsidianHub，当前默认执行方式是：

> **Moss 负责主控、收敛、拆解、审阅、测试与文档；编码执行按任务粒度在 Moss 与外部执行层之间分配；只在跨边界决策时再打断用户。**

这意味着：
- 不再默认每完成一小段就询问方向
- 优先把方向前置收敛成一轮任务包
- 能连续推进时就连续推进
- 真正需要用户判断时再停下

这就是 ObsidianHub 当前最适合的开发协作组织方式。
