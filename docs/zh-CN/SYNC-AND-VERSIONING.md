# ObsidianHub 同步与版本管理

## 1. 核心哲学

ObsidianHub 采用双层模型：

- **实时同步层**：承载“正在进行的工作”
- **Git 版本层**：承载“值得保留的版本节点”

核心原则：

> 实时同步不等于每次保存都创建 Git commit。  
> Git commit 应代表阶段性结果、检查点或有意义的历史节点。

## 2. 为什么要分层

如果把“每次编辑保存”都直接映射成 Git commit，会带来：

- 提交历史过碎
- 大量低价值 commit
- 工作体验笨重
- 同步频率与版本频率强耦合

因此应拆成：

### 实时同步层
负责：
- autosave
- 当前工作状态传播
- 多设备尽快看到最新改动
- 基础冲突提示

### Git 版本层
负责：
- checkpoint
- 历史审计
- 回滚
- 差异比较
- 分支与远程备份基础

## 3. 实时同步层

MVP / MVP+1 阶段建议采用：

- **Obsidian 轻插件**
- **ObsidianHub Sync API**

而不是把 WebDAV 作为核心实时同步机制。

### 轻插件职责
- 服务端连接配置
- 监听本地 vault 文件变化
- debounce 后上传变更
- 拉取远端最新状态
- 显示同步状态
- 显示未提交改动 / 最近同步结果
- 提供手动 sync 操作

### Sync API 职责
- 接收工作态变更
- 维护当前工作副本状态
- 向其他设备提供最新内容
- 维护基础冲突信息
- 与 Git 版本层对接

## 4. Git 版本层

Git 不承担“每打几个字就实时传播”的职责。

Git 主要用于：
- 手动版本保存
- 自动 checkpoint commit
- agent 批处理后的结构化提交
- 历史查看
- 文件/版本恢复
- 可选远程 Git 备份（如 GitHub）

### 适合触发 Git commit 的时机
- 用户手动点击“创建版本”
- 用户执行“同步并提交”
- agent 一轮任务完成
- 批量操作完成
- 创建快照时
- 达到某个稳定 checkpoint 时
- 夜间整理或阶段收束时

## 5. 快照层

快照层是 Git 之外的兜底安全网。

负责：
- 写前备份
- 定时快照
- 单文件恢复
- 整体 vault 恢复

它和 Git 的关系是互补，而不是互相替代。

## 6. MVP 边界

### MVP 支持
- Git 优先的版本管理路线
- 轻量 Obsidian 插件原型
- Sync API 作为工作态同步入口
- 基础的 Git checkpoint 能力
- 快照与恢复

### MVP 不做
- 重型实时协作编辑
- CRDT / OT
- 复杂 merge UI
- 多人光标级协同
- 复杂多分支协作工作流

## 7. 产品体验目标

理想体验是：

- 像实时文档一样顺滑地工作
- 像 Git 一样可靠地管理版本
- 像控制平面一样可观察、可恢复、可审计
